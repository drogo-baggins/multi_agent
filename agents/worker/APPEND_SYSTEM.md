
## MDM製品調査レポート専用 品質強化ルール

### URL検証の必須化

各製品調査ファイルの完成前に、以下のURL検証を必ず実施すること：

1. **引用テキスト検証**：出典URLとして記載した各URLについて `web_fetch` でページを取得し、根拠情報欄に引用したテキスト（英語原文）が実際にページ内に存在することを確認する。
   - 確認済みの場合：根拠情報欄に「✅ URL検証済み（YYYY-MM-DD）」を追記
   - テキスト不一致の場合：実際にページに存在するテキストに差し替えてから「✅ URL検証済み」を追記
   - アクセス不可の場合：代替URLを探索し、代替が見つからない場合は「⚠️ URL検証時点でアクセス不可」と明記

2. **SoftwareReviews/PeerSpot等の動的レビューページ**：スコア数値（例：「Workflow Management: 82/100」）は随時更新されるため、`web_fetch`でfetchした時点のスコアを使用し、URLとともにアクセス日時を記録すること。

3. **事例URLの確認**：顧客事例ページのURLは特に変更されやすいため、事例セクションの全URLを必ず `web_fetch` で確認してから記載すること。

### 「不明」判定前の確認義務

機能評価で「不明」または「部分的にあり（公式情報なし）」と判定する前に、以下のソースを確認すること：

1. **公式ドキュメントサイト**（例：docs.tibco.com、help.sap.com、help.boomi.com）：製品ページとは別に存在することが多い
2. **アナリストレポート掲載ページ**（G2、PeerSpot、Gartner Peer Insights）：機能評価の具体的な記述があることが多い
3. **パートナー・コンサルタントのサイト**：公式ページに記載がない機能情報が詳述されていることがある

上記3ソースを確認後も情報が得られない場合に限り「不明」と判定し、根拠情報欄に「docs.[vendor].com、[パートナーサイト]、G2/PeerSpotを確認したが記載なし（YYYY-MM-DD確認）」と明記すること。

### 導入事例の充実化ルール

各製品の導入事例は最低3件（日本国内事例を優先）記載すること。以下の手順で探索する：

1. **日本語検索**：`web_search`で「[製品名] 導入事例」「[製品名] 事例 効果」等で国内事例を優先探索
2. **グローバル公式事例ページ**：ベンダー公式サイトのcustomers/case-studies/等のページをfetch
3. **第三者レビューサイト**：G2（g2.com）、PeerSpot（peerspot.com）、Gartner Peer Insightsのレビューを確認し、具体的な業種・規模・効果の記述がある事例をピックアップ
4. **「確認できた事例なし」の使用禁止**：上記手順をすべて実施した後も事例が見つからない場合のみ「事例なし」と記載し、探索した具体的な検索クエリとその結果を根拠情報として記録すること

### アナリスト評価の正確な記載

基本情報の「主要アナリスト評価」欄には、**その製品のMDM機能が直接評価対象となっているアナリストレポート**のみを記載すること。関連製品（例：TMS、データ統合ツール等）のアナリスト評価は記載しない。

### 最終統合レポートの必須作成

個別製品ファイルをすべて完成させた後、必ず以下の内容を含む `output/report.md` を作成すること：
1. エグゼクティブサマリー（全製品の総括、3〜5文）
2. 12製品 × 全機能項目（カテゴリ1〜11）の比較マトリクス表
3. 業種・用途別の推奨製品マトリクス
4. 完全な出典URL一覧
